## 2. Real BR listings table found

The real Bienes Raices listings table is `public.listings`.

BR uses shared `public.listings`, not a dedicated real-estate table.

Relevant existing fields:

- `id`: real listing row id.
- `leonix_ad_id`: public Leonix Ad ID.
- `owner_id`: owner/agent/business identity fallback.
- `category`: `bienes-raices` for BR rows.
- `seller_type`: `business` for Negocio, `personal` for Privado.
- `status` and `is_published`: public lifecycle flags; active public rows use `status = 'active'` and `is_published is true`.
- `detail_pairs`: legacy human and machine contract array.
- `listing_json`, `profile_json`, `contact_json`: Gate 12D-2 structured JSONB columns.
- `business_name`, `business_meta`: business identity/profile fields.

Property type/subcategory is stored in `detail_pairs` via `Leonix:categoria_propiedad`, `Leonix:property_subtype`, and `Leonix:results_property_kind`.

Live detail URL is `/clasificados/anuncio/[id]`, generated by `leonixLiveAnuncioPath`. Dashboard rows come from `app/(site)/dashboard/lib/ownerListingsQuery.ts`. Admin BR rows come from `app/admin/_lib/listingsAdminSelect.ts` through the category queue. No current BR inventory grouping query or BR Negocio active listing limit was found.

## 3. Existing BR publish/detail foundation summary

BR Privado and BR Negocio publish through `publishLeonixRealEstateListingCore`, which inserts real rows into `public.listings`. The live detail route reads `public.listings` by id, displays `leonix_ad_id`, hydrates `detail_pairs` from the structured JSONB columns when present, and renders the shared BR/En Venta detail layout.

## 4. Data model decision

SQL needed: TRUE.

Chosen model: add real nullable grouping columns to `public.listings`:

- `br_inventory_group_id uuid null`
- `br_inventory_parent_listing_id uuid null`
- `inventory_role text null`

Allowed role values are `main` and `inventory_property`. Existing listings stay valid because all columns are nullable. Every property remains a real listing row with its own `id`, `leonix_ad_id`, and detail URL. No nested property payload was introduced.

Fallback rule: prefer `br_inventory_group_id`; if missing, use `owner:{owner_id}` for BR Negocio grouping.

## 9. Verification SQL

```sql
-- BR13A verification SQL.
-- Run in Supabase only after applying the BR property inventory migration.

with br_negocio as (
  select
    l.id,
    l.leonix_ad_id,
    l.title,
    l.status,
    l.is_published,
    l.owner_id,
    l.business_name,
    l.seller_type,
    l.br_inventory_group_id,
    l.br_inventory_parent_listing_id,
    l.inventory_role,
    l.created_at,
    l.updated_at,
    l.published_at,
    coalesce(l.br_inventory_group_id::text, 'owner:' || l.owner_id::text) as inventory_count_key
  from public.listings l
  where l.category = 'bienes-raices'
    and (
      l.seller_type = 'business'
      or l.detail_pairs @> '[{"label":"Leonix:branch","value":"bienes_raices_negocio"}]'::jsonb
    )
),
active_counts as (
  select
    inventory_count_key,
    count(*) filter (
      where status = 'active'
        and is_published is true
    ) as active_negocio_property_count
  from br_negocio
  group by inventory_count_key
)
select
  b.id,
  b.leonix_ad_id,
  b.title,
  b.status,
  b.is_published,
  b.owner_id,
  b.business_name,
  b.seller_type,
  b.br_inventory_group_id,
  b.br_inventory_parent_listing_id,
  b.inventory_role,
  b.created_at,
  b.updated_at,
  b.published_at,
  a.active_negocio_property_count,
  5 as recommended_active_property_limit
from br_negocio b
left join active_counts a
  on a.inventory_count_key = b.inventory_count_key
order by coalesce(b.published_at, b.updated_at, b.created_at) desc nulls last
limit 50;
```
